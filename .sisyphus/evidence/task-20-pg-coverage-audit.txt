FINAL ASSESSMENT
═══════════════════════════════════════════════════════════════════════

Of 29 memory test files:
  ✅ 22 FULLY COVERED by existing PG tests
  ⚡  6 PARTIALLY covered (storage layer covered; orchestration is above-storage)
  ❌  1 SQLite-only (schema.test.ts) — PG schema tests already exist separately

Of 20 PG domain repos:
  ✅ 16 have dedicated PG integration tests
  🔶  4 have no dedicated tests (all read-query repos)

RECOMMENDED NEW TEST: 1 file
  → pg-narrative-search-repo.test.ts
    Reason: PgNarrativeSearchRepo uses pg_trgm similarity/word_similarity/GREATEST/ILIKE
    which are PG-specific functions not exercised by any other test. This is the
    only genuine storage-behavior gap where PG query semantics differ from SQLite FTS5.

SKIPPED (acceptable gaps):
  → PgPromotionQueryRepo: Read-only, pure-logic decision functions, tested indirectly
  → PgGraphReadQueryRepo: Read-only over validated tables, massive setup for marginal gain
  → PgNodeScoringQueryRepo: Read-only scoring queries, tables already validated

CONCLUSION:
PG integration test coverage is STRONG. 22 of 29 memory behaviors are fully covered.
The 6 "partial" cases are above-storage orchestration logic that will be addressed in T27
when SQLite service wrappers are replaced. The ONE genuine gap is PgNarrativeSearchRepo
which exercises PG-specific trgm functions not tested elsewhere.
