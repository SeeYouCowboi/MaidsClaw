# Area/World State Events Schema

## Context

This document defines the schema for `area_state_events` and `world_state_events` — the append-only event log tables that back the `area_state_current` and `world_state_current` projection tables.

The design follows the temporal projection consensus from §18.13, §18.15, and §18.20 of `docs/MEMORY_REFACTOR_CONSENSUS_PLAN_2026-03-20.zh-CN.md`:

- **Two formal time axes**: `valid_time` (when the state became true in the world) and `committed_time` (when the system recorded it). These must never be conflated.
- **Hybrid model**: current projections for fast reads, event log for time-slice queries and replay. Neither pure current-only nor pure replay-only.
- **Area state is the authoritative backend ontology layer** (`area_state_current`), not a narrative surface. Latent states can exist here before any agent perceives them.
- **World state has a higher entry bar** than area state — only stable, public, externalized facts belong here.

The events tables are the source of truth. The `_current` tables are derived projections that can be rebuilt from the event log at any time.

## Schema: area_state_events

### CREATE TABLE SQL

```sql
CREATE TABLE IF NOT EXISTS area_state_events (
  id               INTEGER PRIMARY KEY,
  agent_id         TEXT    NOT NULL,
  area_id          INTEGER NOT NULL,
  key              TEXT    NOT NULL,
  value_json       TEXT    NOT NULL,
  surfacing_classification TEXT NOT NULL
    CHECK (surfacing_classification IN (
      'public_manifestation',
      'latent_state_update',
      'private_only'
    )),
  source_type      TEXT    NOT NULL DEFAULT 'system'
    CHECK (source_type IN (
      'system',
      'gm',
      'simulation',
      'inferred_world'
    )),
  valid_time       INTEGER,
  committed_time   INTEGER NOT NULL,
  settlement_id    TEXT    NOT NULL,
  created_at       INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_area_state_events_agent_area_key
  ON area_state_events(agent_id, area_id, key, committed_time DESC);

CREATE INDEX IF NOT EXISTS idx_area_state_events_settlement
  ON area_state_events(settlement_id);

CREATE INDEX IF NOT EXISTS idx_area_state_events_valid_time
  ON area_state_events(agent_id, area_id, valid_time DESC);
```

### Column Details

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | INTEGER | NOT NULL | autoincrement | Surrogate primary key. |
| `agent_id` | TEXT | NOT NULL | — | The agent whose area-scoped state this event belongs to. Matches `area_state_current.agent_id`. |
| `area_id` | INTEGER | NOT NULL | — | The area this state key belongs to. Matches `area_state_current.area_id`. |
| `key` | TEXT | NOT NULL | — | State key within the area. Matches `area_state_current.key`. |
| `value_json` | TEXT | NOT NULL | — | JSON-serialized state value at the time of this event. |
| `surfacing_classification` | TEXT | NOT NULL | — | Visibility classification. `public_manifestation` = publicly perceptible; `latent_state_update` = backend state, not auto-surfaced to narrative; `private_only` = stays in private episode/cognition only. |
| `source_type` | TEXT | NOT NULL | `'system'` | Provenance of this state change. `system` = runtime-generated; `gm` = game-master override; `simulation` = world simulation step; `inferred_world` = inferred from world-level reasoning. |
| `valid_time` | INTEGER | NULL | — | Unix ms timestamp of when this state became true in the world. NULL means unknown or same as committed_time. Supports late-evidence and retroactive correction semantics. |
| `committed_time` | INTEGER | NOT NULL | — | Unix ms timestamp of when this event was written into the system (settlement time). Always set. |
| `settlement_id` | TEXT | NOT NULL | — | The settlement batch this event belongs to. Links events to the interaction/publication cycle that produced them. |
| `created_at` | INTEGER | NOT NULL | — | Unix ms timestamp of row insertion. Monotonically increasing. |

#### surfacing_classification values

- `public_manifestation`: The state change has a publicly perceptible manifestation in the area (audible speech, visible action, obvious environmental change). Eligible for automatic materialization into narrative.
- `latent_state_update`: Updates the authoritative backend area state but does not automatically generate a narrative event. Examples: a door is now locked, a drawer contains a letter, a room has a gas leak.
- `private_only`: Stays in private episode or private cognition. Not externalized to area or world layers.

#### source_type values

- `system`: Generated by the runtime during normal agent operation.
- `gm`: Injected by a game-master or operator override.
- `simulation`: Produced by a world simulation step (e.g., environmental decay, time-based state change).
- `inferred_world`: Derived from world-level reasoning or inference, not from a direct agent action.

## Schema: world_state_events

### CREATE TABLE SQL

```sql
CREATE TABLE IF NOT EXISTS world_state_events (
  id               INTEGER PRIMARY KEY,
  key              TEXT    NOT NULL,
  value_json       TEXT    NOT NULL,
  surfacing_classification TEXT NOT NULL
    CHECK (surfacing_classification IN (
      'public_manifestation',
      'latent_state_update',
      'private_only'
    )),
  source_type      TEXT    NOT NULL DEFAULT 'system'
    CHECK (source_type IN (
      'system',
      'gm',
      'simulation',
      'inferred_world'
    )),
  valid_time       INTEGER,
  committed_time   INTEGER NOT NULL,
  settlement_id    TEXT    NOT NULL,
  created_at       INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_world_state_events_key_committed
  ON world_state_events(key, committed_time DESC);

CREATE INDEX IF NOT EXISTS idx_world_state_events_settlement
  ON world_state_events(settlement_id);

CREATE INDEX IF NOT EXISTS idx_world_state_events_valid_time
  ON world_state_events(key, valid_time DESC);
```

### Column Details

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | INTEGER | NOT NULL | autoincrement | Surrogate primary key. |
| `key` | TEXT | NOT NULL | — | World-scoped state key. Matches `world_state_current.key`. |
| `value_json` | TEXT | NOT NULL | — | JSON-serialized state value at the time of this event. |
| `surfacing_classification` | TEXT | NOT NULL | — | Same classification as area events. World-level entries should predominantly be `public_manifestation` given the higher entry bar for world state. |
| `source_type` | TEXT | NOT NULL | `'system'` | Same provenance enum as area events. |
| `valid_time` | INTEGER | NULL | — | When this world state became true. NULL = unknown or same as committed_time. |
| `committed_time` | INTEGER | NOT NULL | — | When this event was written into the system. |
| `settlement_id` | TEXT | NOT NULL | — | Settlement batch this event belongs to. |
| `created_at` | INTEGER | NOT NULL | — | Row insertion timestamp. |

**Note on world_state_events vs area_state_events**: World state has no `agent_id` or `area_id` because it is global. The entry bar is higher — world state should only contain stable, public, externalized facts (§18.21). Latent states belong in area state, not world state.

## Relationship: events ↔ current

```
area_state_events                    area_state_current
─────────────────────────────────    ──────────────────────────────────────
(agent_id, area_id, key, ...)   →    PRIMARY KEY (agent_id, area_id, key)
committed_time (latest wins)    →    committed_time
valid_time                      →    valid_time
value_json                      →    value_json
surfacing_classification        →    surfacing_classification
source_type                     →    source_type
committed_time                  →    updated_at  (set to committed_time on upsert)

world_state_events                   world_state_current
─────────────────────────────────    ──────────────────────────────────────
(key, ...)                      →    PRIMARY KEY (key)
committed_time (latest wins)    →    committed_time
valid_time                      →    valid_time
value_json                      →    value_json
surfacing_classification        →    surfacing_classification
committed_time                  →    updated_at
```

The `_current` tables hold exactly one row per `(agent_id, area_id, key)` or `key`. They always reflect the state from the event with the highest `committed_time` for that key. The events tables hold the full history.

**Current table columns not in events tables**: `updated_at` in `_current` is a denormalized copy of `committed_time` from the winning event, kept for backward-compatible index queries (`idx_area_state_current_agent_area` sorts by `updated_at DESC`).

**Columns in events tables not in current tables**: `settlement_id` and `created_at` are event-log metadata. They don't belong in the current projection.

## Replay Rules

Replay means rebuilding `_current` tables from scratch using only the events tables. The rules are:

### area_state_current rebuild

```sql
-- For each (agent_id, area_id, key), take the event with the highest committed_time.
-- If two events share the same committed_time, take the one with the higher id (insertion order).

INSERT INTO area_state_current (
  agent_id, area_id, key, value_json,
  surfacing_classification, source_type,
  updated_at, valid_time, committed_time
)
SELECT
  agent_id, area_id, key, value_json,
  surfacing_classification, source_type,
  committed_time AS updated_at,
  valid_time,
  committed_time
FROM area_state_events e1
WHERE e1.id = (
  SELECT id FROM area_state_events e2
  WHERE e2.agent_id = e1.agent_id
    AND e2.area_id  = e1.area_id
    AND e2.key      = e1.key
  ORDER BY e2.committed_time DESC, e2.id DESC
  LIMIT 1
);
```

### world_state_current rebuild

```sql
-- For each key, take the event with the highest committed_time.

INSERT INTO world_state_current (
  key, value_json, surfacing_classification,
  updated_at, valid_time, committed_time
)
SELECT
  key, value_json, surfacing_classification,
  committed_time AS updated_at,
  valid_time,
  committed_time
FROM world_state_events e1
WHERE e1.id = (
  SELECT id FROM world_state_events e2
  WHERE e2.key = e1.key
  ORDER BY e2.committed_time DESC, e2.id DESC
  LIMIT 1
);
```

### Time-slice query (point-in-time read)

To answer "what was the area state at time T?" use `valid_time` for world-truth queries and `committed_time` for agent-knowledge queries (§18.15):

```sql
-- World-truth at valid_time T: what was actually true in the world at time T?
SELECT agent_id, area_id, key, value_json, surfacing_classification, source_type, valid_time, committed_time
FROM area_state_events e1
WHERE valid_time <= :T
  AND e1.id = (
    SELECT id FROM area_state_events e2
    WHERE e2.agent_id = e1.agent_id
      AND e2.area_id  = e1.area_id
      AND e2.key      = e1.key
      AND e2.valid_time <= :T
    ORDER BY e2.valid_time DESC, e2.id DESC
    LIMIT 1
  );

-- Agent-knowledge at committed_time T: what did the system know at time T?
-- Replace valid_time with committed_time in the above query.
```

### Replay trigger conditions

Replay should be triggered when:

1. **Schema migration** adds or changes columns in `_current` tables.
2. **Data correction** — a past event is found to be wrong and a corrective event is appended.
3. **Disaster recovery** — the `_current` table is lost or corrupted.
4. **Audit** — verifying that `_current` matches the event log.

Replay is never triggered during normal runtime operation. The `_current` tables are maintained incrementally via upsert on each new event.

---

## Append-only Protection

The events tables are append-only. No UPDATE or DELETE is permitted on committed rows.

**Correction semantics**: If a past state was wrong, append a new event with:
- `valid_time` set to the time the corrected state actually became true (may be in the past).
- `committed_time` set to now (when the correction is being recorded).
- `settlement_id` referencing the correction settlement.

This preserves the full audit trail and supports retroactive correction without mutating history (§18.15: "late evidence / retroactive correction semantics").

**Enforcement**: The append-only constraint is enforced at the application layer (no UPDATE/DELETE queries in the repo). A future migration may add a SQLite trigger to enforce this at the database layer:

```sql
CREATE TRIGGER area_state_events_no_update
BEFORE UPDATE ON area_state_events
BEGIN
  SELECT RAISE(ABORT, 'area_state_events is append-only');
END;

CREATE TRIGGER area_state_events_no_delete
BEFORE DELETE ON area_state_events
BEGIN
  SELECT RAISE(ABORT, 'area_state_events is append-only');
END;
```

The proposed schema was validated against the existing `area_state_current` DDL in `src/memory/schema.ts` (migration 015, 020, 023) and the upsert logic in `src/memory/projection/area-world-projection-repo.ts`.

### Observations

1. **Column alignment confirmed**: `area_state_events` columns map directly to `area_state_current` columns. The events table adds `settlement_id` and `created_at`; the current table adds `updated_at` (derived from `committed_time`).

2. **world_state_current gap**: The existing `world_state_current` does not have `source_type`. The `world_state_events` table includes `source_type` for provenance tracking. When T11 implements the migration, `world_state_current` should also gain `source_type` to stay in sync with its events table.

3. **world_state_current upsert gap**: The existing `upsertWorldStateCurrent` in `area-world-projection-repo.ts` (lines 149-157) does not write `valid_time`, `committed_time`, or `source_type`. T11 must update this upsert when adding the events table, so the current projection stays consistent.

4. **committed_time nullable in current**: `committed_time` is nullable in `area_state_current` and `world_state_current` (legacy rows may have NULL). In `area_state_events` and `world_state_events`, `committed_time` is NOT NULL — every new event must have a committed time.

5. **settlement_id**: Not present in any existing table. This is a new column introduced by the events schema to link events to their originating settlement batch.
