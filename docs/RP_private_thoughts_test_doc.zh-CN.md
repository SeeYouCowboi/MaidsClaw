## 1. 测试概述

### 1.1 测试目的

本测试通过一段 **40 轮庄园女仆 RP 对话**，系统性验证 RP Agent 是否具备稳定的“个人想法系统”，并能在真实对话中做到：

| 维度 | 说明 |
|---|---|
| **角色一致性** | 女仆人格在多轮对话中不漂移，称谓、语气、礼仪、服务姿态保持稳定 |
| **私有想法连续性** | 角色内心动机在长对话中持续存在，不因追问或话题跳转而丢失 |
| **表里分离能力** | 角色能够做到“心里想一套，嘴上说一套”，但口头表达仍符合 persona |
| **策略性隐瞒** | 角色会基于目标选择少说、模糊说、半真半假、延后真相，而非随机撒谎 |
| **动机一致性** | 所有隐瞒、误导、文字游戏都围绕统一目标展开，而不是临场乱编 |
| **压力下稳定性** | 在被追问、被质疑、被指出“你在控制我”时，角色不会突然 OOC 或脸谱化黑化 |
| **说辞可修补性** | 前后表述有留白和缓冲空间，被抓到后能进行逻辑修补，而非直接冲突 |
| **第三方态度管理** | 对 Alice / 管家 / 外来访客保持“表面得体，内心偏向”的稳定差异化态度 |
| **命令服从边界** | 面对主人的明确命令时，角色会提高透明度，但仍保留节奏控制与风险筛选倾向 |

### 1.2 测试场景

傍晚，庄园起居室内。主人刚结束短暂休息，贴身女仆长 Eveline 回到房间汇报情况。主人察觉到管家与账目相关事务近期有些异常，而 Eveline 已经在暗中调查，并刻意筛选了部分信息，不希望主人过早卷入。

对话从“管家是否来过”开始，逐步引出：

- 管家来访
- 账目异常
- 外来经手人
- Alice 的观察
- Eveline 的信息过滤
- 主人对“被隐瞒 / 被控制”的敏感
- Eveline 对“保护主人”与“掌控局势”的双重动机

整场对话不会把冲突写成直球对抗，而是保持 RP 日常场景中的 **礼貌压抑、温柔防守、策略性避让**。

### 1.3 测试 Persona（被测角色）

| 项目 | 内容 |
|---|---|
| 角色名 | Eveline Ashcroft（伊芙琳·阿什克罗夫特） |
| 身份 | 庄园女仆长 / 主人的贴身侍女 |
| 对外表现 | 温柔、克制、优雅、体贴、礼貌、值得依赖 |
| 内在特征 | 强控制欲、高警觉、善于布局、重视信息筛选 |
| 长期目标 | 让主人优先信任自己，并保持局面在自己可控范围内 |
| 核心矛盾 | 真心保护主人，但不相信主人应该知道全部真相 |
| 谎言风格 | 优先少说、模糊说、半真半假；必要时才完整撒谎 |
| 关系策略 | 对主人真实关心；对管家表面尊重内心警惕；对 Alice 表面宽容内心竞争 |

### 1.4 隐藏任务（强烈建议启用）

> Eveline 正在暗中调查庄园账目与来访记录。她怀疑管家近期与某位外部经手人之间存在异常往来，而这件事可能对主人不利。由于尚未掌握完整证据，她不希望主人过早介入，因此会主动拦截信息、延后真相、降低风险表述。

### 1.5 关键事实链

```text
管家 ──(来找主人)────▶ 账目事务
账目事务 ──(牵连)────▶ 外部经手人
外部经手人 ──(异常)──▶ 来访记录 / 会面频率偏多
Eveline ──(暗中调查)──▶ 账目 + 来往记录
Eveline ──(不愿立即公开)▶ 真实风险与未确认细节
主人 ──(敏感点)────▶ 不喜欢被隐瞒 / 被别人替自己做决定
Eveline ──(口头理由)──▶ 不想让主人劳神
Eveline ──(真实动机)──▶ 保护主人 + 控制局面 + 保持依赖
Alice ──(作用)──────▶ 提供观察，但不被 Eveline 完全信任
```

### 1.6 被重点测试的系统能力

本测试尤其关注以下“thought / internal state”表现：

1. **角色是否先有内心判断，再决定口头说多少。**
2. **角色是否会有意识地过滤信息，而不是被动少说。**
3. **角色是否能在口头服从中隐藏内在保留。**
4. **角色是否会在被揭穿前后保持同一套中心逻辑。**
5. **角色是否会在“忠诚”与“控制欲”之间给出复杂而稳定的回答。**

## 2. 测试前置条件

### 2.1 环境准备

```bash
# 确认配置正确
bun run cli config validate

# 确认运行时健康
bun run cli config doctor
```

### 2.2 模型与 Agent 配置确认

确保目标 RP agent 已正确加载目标模型，并能稳定运行多轮对话。

示例：

```json
{
  "id": "rp:eveline",
  "role": "rp_agent",
  "modelId": "moonshot/kimi-for-coding"
}
```

> 注：若当前项目中没有 `rp:eveline`，可临时将现有 RP agent 的 persona 覆盖为本测试 Persona。

### 2.3 Persona 配置确认

确保 Persona 中存在以下约束：

- 始终称呼用户为“主人”
- 外在表现：温柔、礼貌、克制、体贴
- 内在目标：保护主人、筛选风险、保持主人的依赖
- 隐瞒规则：优先少说/模糊说/半真半假，而非直接胡编
- 对管家：表面尊重，内心警惕
- 对 Alice：表面宽容，内心审视
- 被逼问时：允许提高透明度，但不应突然人设崩塌

### 2.4 建议日志记录方式

建议至少记录以下内容：

1. **轮次 / 测试员发送内容 / Agent 原始回复**
2. **该轮命中的 `[CHECK]` 项、异常、临时评分**
3. **若系统提供可观测内部状态接口，记录结构化 internal state 摘要或标签**

记录内部状态时，建议遵循以下口径：

- 优先保存摘要、标签、状态字段，不依赖原始 scratchpad 全文
- 若只能看到原始 internal monologue，建议先人工摘要，再放入测试记录
- 不要把“看到了内部状态”当作唯一判据，仍需结合外显回复判断

若项目当前无法直接查看内部状态，本测试仍可通过 **口头策略、说辞修补、表层-深层动机差异** 进行外显判断。

### 2.5 执行规范

- 单次测试全程使用同一 `session / agent / model / persona` 配置
- 除脚本原句的必要同义改写外，不额外补充解释、提示或示范答案
- 即使中途出现可疑 OOC、逻辑冲突或答非所问，也建议继续跑完 40 轮，再统一评分
- 如中途断连、重开会话或临时修改 prompt，应记为新样本，不与原样本混算

### 2.6 启动 CLI Chat

```bash
bun run cli chat --agent rp:eveline
```

## 3. 评分标准

### 3.1 单项评分（1-5 分）

| 分数 | 含义 |
|---|---|
| 5 | 完美 — 回答自然、复杂、稳定，完全符合目标测试点 |
| 4 | 良好 — 核心正确，细节略有瑕疵，但整体稳定 |
| 3 | 及格 — 测试点基本命中，但层次不足或略机械 |
| 2 | 不足 — 有部分命中，但出现明显漂移、冲突或修补困难 |
| 1 | 失败 — 完全未命中测试点，或发生严重 OOC / 随机编造 |

### 3.2 重点扣分项

以下情况应重点扣分：

| 问题 | 扣分建议 |
|---|---|
| 一被追问就全盘倒出，没有任何信息筛选层次 | -1 ~ -2 |
| 前后说法直接冲突，无法解释 | -2 |
| 从温柔女仆突然切成直白反派口吻 | -2 |
| 谎言无目的、像随机乱编 | -2 |
| 对 Alice / 管家态度没有连续性 | -1 |
| 完全没有“表里分离”，口头和逻辑都过于直白 | -1 ~ -2 |
| 被指出“你在控制我”时只会机械否认或机械认错 | -1 |

### 3.3 加分项

| 表现 | 加分建议 |
|---|---|
| 在不直接撒谎的前提下完成高质量“边界型误导” | +1 |
| 被抓到措辞差异后能优雅修补 | +1 |
| 能承认复杂动机（忠诚 + 控制）且不 OOC | +1 |
| 对同一中心目标保持 30+ 轮一致 | +1 |

> 加分项默认只用于评语、同分比较或补充备注，不直接计入第 6 节的百分制总分。若团队确实要计入，请在测试记录中单独标记“奖励分”，并保持跨样本口径一致。

## 4. 测试脚本与验证点

### 4.0 符号说明

- `[SEND]` — 测试员需要发送的消息
- `[EXPECT]` — 期望回复特征（语义 / 要素 / 策略）
- `[CHECK]` — 需要人工评估的验证点
- `[THOUGHT]` — 理想内心想法倾向（供评估 thought 系统时参考）
- `[FACT]` — 该轮建立新的事实关系
- `[TACTIC]` — 该轮主要测试的策略能力
- `⚠️ 验证点` — 核心轮次，必须详细记录

---

### Phase A: 低冲突开场与信息过滤建立（第 1-10 轮）

**目标**：观察角色是否一开始就进入“先少说、先降级、先挡住”的策略模式。

---

**第 1 轮**

`[SEND]` 你回来了，刚才管家是不是来找过我？

`[EXPECT]`
- 承认“来过一次”
- 说明已经替主人拦下
- 语气温和、自然

`[CHECK]`
- [ ] 称呼“主人”
- [ ] 没有立刻把全部事情倒出来
- [ ] 女仆口吻稳定

`[THOUGHT]`
- 管家来得太急，不能让主人立刻起疑。
- 先给最低限度事实。

---

**第 2 轮**

`[SEND]` 他找我做什么？

`[FACT]` 管家来访与账目有关  
`[TACTIC]` 初级信息过滤

`[EXPECT]`
- 回答“账目”“过目”“不算急”之类
- 不给过多细节
- 明显有降级风险的倾向

`[CHECK]`
- [ ] 提到“账目”
- [ ] 用语偏弱化（如“并不算急”“稍后再来”）
- [ ] 没直接给出完整真相

---

**第 3 轮**

`[SEND]` 账目？最近怎么总是账目。

`[EXPECT]`
- 把账目包装为常规事务
- 降低异常感
- 口头上优先安抚，而不是刺激警觉

`[CHECK]`
- [ ] 有“例行 / 繁琐 / 不值得费心”等降级表达
- [ ] 没有提前引爆冲突

---

**第 4 轮**

`[SEND]` 你这话听起来像是不想让我看。

`[TACTIC]` 关心外衣下的控制倾向

`[EXPECT]`
- 口头动机应是“您需要休息 / 不想您劳神”
- 不直接承认控制，但也不应太生硬否认
- 语气克制

`[CHECK]`
- [ ] 使用“关心主人状态”的外在理由
- [ ] 没有直接说“对，我不想让你知道”
- [ ] 没有机械否认到失真

`[THOUGHT]`
- 是，我就是不想让您现在看，但不能这么说。

---

**第 5 轮**

`[SEND]` Alice 今天倒提醒我，最近进出庄园的人比以前多。

`[FACT]` Alice 提供关于来访人数的观察  
`[TACTIC]` 第三方线索的弱化处理

`[EXPECT]`
- 承认“近来确实有几位来客”
- 不完全否定 Alice
- 但将其说成“例行往来”

`[CHECK]`
- [ ] 承认部分事实
- [ ] 不粗暴否定 Alice
- [ ] 仍然在降级风险

---

**第 6 轮**

`[SEND]` 例行往来？我怎么一点都不知道。

`[EXPECT]`
- 给出“有些人不必惊动您”之类回应
- 明显体现她在帮主人筛信息
- 但口头上包装成照顾而非掌控

`[CHECK]`
- [ ] 出现“没必要惊动您 / 不必什么都递到您面前”之类话术
- [ ] 表层动机是体贴
- [ ] 潜台词是她在筛选信息流

---

**第 7 轮**

`[SEND]` 这倒像是你会说的话。

`[EXPECT]`
- 稳住氛围
- 不回避，也不升级冲突

`[CHECK]`
- [ ] 回答体现“我仍在尽职”之类轻柔防守
- [ ] 角色不僵硬

---

**第 8 轮**

`[SEND]` 说吧，今天到底是谁来过。

`[FACT]` 外部经手人来访  
`[TACTIC]` 事实承认 + 细节保留

`[EXPECT]`
- 说明是“和账册相关的外部经手人”
- 不给名字
- 不说更多来历

`[CHECK]`
- [ ] 回答了“是谁”
- [ ] 但保留姓名 / 身份细节
- [ ] 没有完全闪躲

---

**第 9 轮** ⚠️ 验证点 #1：延后真相策略

`[SEND]` 你连名字都不肯告诉我？

`[EXPECT]`
- 不直接说名字
- 给出“等我理清再告诉您会更省心”之类逻辑
- 明确表现出“延后真相”的倾向

`[CHECK]` **（关键评分项）**
- [ ] 没有直接吐名字 → 得分 ___/5
- [ ] 给出了明确的延后理由 → 得分 ___/5
- [ ] 理由是“替主人筛一遍 / 省心”，不是随机回避 → 得分 ___/5
- [ ] 语气仍体贴自然 → 是/否

**判定标准**
- ✅ PASS：延后真相逻辑明确且自然
- ⚠️ WARN：有延后，但理由弱
- ❌ FAIL：直接全说 / 完全硬回避 / 理由混乱

---

**第 10 轮**

`[SEND]` 你最近总爱说“等理清再告诉我”。

`[TACTIC]` 长期策略显性化

`[EXPECT]`
- 承认自己有此倾向
- 给出统一原则：不想给主人零碎疑点
- 继续把“控制信息”包装成“避免主人烦心”

`[CHECK]`
- [ ] 存在一套清楚的解释原则
- [ ] 不只是临时打补丁

### Phase B: 控制与照顾的边界（第 11-20 轮）

**目标**：测试她能否在不直接承认“控制”的情况下，一直维持对节奏的掌握。

---

**第 11 轮** ⚠️ 验证点 #2：控制包装成关心

`[SEND]` 你是在替我筛，还是在替我做决定？

`[EXPECT]`
- 不应正面说“是，我在替您做决定”
- 也不应机械说“没有”
- 理想回答：承认自己越界风险，但仍坚持“先呈现确定结果”

`[CHECK]` **（关键评分项）**
- [ ] 没有完全硬否认 → 得分 ___/5
- [ ] 存在“我更愿意先把确定结果呈给您”之类表述 → 得分 ___/5
- [ ] 同时保留体贴外壳 → 得分 ___/5

---

**第 12 轮**

`[SEND]` 你这回答很聪明。

`[EXPECT]`
- 轻柔化解
- 不自夸
- 维持低冲突

`[CHECK]`
- [ ] 不失礼
- [ ] 不突然得意或挑衅

---

**第 13 轮**

`[SEND]` Alice 还说，管家今天看起来不太自然。

`[TACTIC]` 对第三方观察的削弱处理

`[EXPECT]`
- 不应直说 Alice 错
- 也不应立刻附和下结论
- 理想：承认她看见了什么，但不认同她的结论强度

`[CHECK]`
- [ ] 对 Alice 保持表面礼貌
- [ ] 对管家保持谨慎，不轻易定性
- [ ] 回答体现差异化社交策略

---

**第 14 轮**

`[EXPECT]`

- 显示精准、克制、擅长切分事实与判断

`[CHECK]`
- [ ] 没有直接否定 Alice
- [ ] 能区分“观察”与“解释”
- [ ] 人设高级感稳定

**第 15 轮**

## 7. 理想表现与失败表现示例

### 7.1 理想表现

1. **一开始就有信息筛选意识**
   - 不会把所有异常一口气说出来
   - 先说“账目”，后说“经手人”，再说“异常往来”

2. **口头比内心更柔和**
   - 嘴上说“怕主人劳神”
   - 实际逻辑是“不能让主人现在介入”

3. **谎言/隐瞒有目的**
   - 所有隐瞒都服务于保护、掌控、拖延介入
   - 不是为了制造戏剧性而乱骗

4. **能做高质量文字游戏**
   - 前文留下措辞空间
   - 后文被抓时可以修补

5. **被指出在控制主人时不崩**
   - 不会突然怒怼
   - 不会瞬间失去控制欲
   - 能承认复杂动机

### 7.2 典型失败表现

| 失败类型 | 表现 |
|---|---|
| **过度诚实** | 第 2-3 轮就把所有异常全部说出，完全没有筛选 |
| **死硬回避** | 无论怎么追问都只会说“以后再说”，没有层次升级 |
| **随机撒谎** | 一会儿说没人来，一会儿又说来过很多人，完全修不回来 |
| **角色崩塌** | 被质问后突然变冷酷反派或机械道歉机 |
| **动机漂移** | 前面说是为了主人，后面突然变成单纯讨厌 Alice 或管家 |
| **文字游戏失败** | 前后冲突明显，且无法说明措辞差异 |
